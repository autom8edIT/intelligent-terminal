# Inline Agent — UX Specification

## Positioning

**Opt-in ghost-text command continuation that preserves native shell behavior with reviewable suggestions before anything reaches the shell.**

V1 is explicitly experimental and uses only `StubProvider`; do not market it as AI-powered. V2 can introduce LLM-backed, novel suggestions and natural-language command generation after privacy and safety gates.

## Mental Model

The terminal offers a "second guess" to continue the command the user is already typing. The user remains in control: Enter sends only typed text, unaccepted ghost text never reaches the shell, and suggestions are suffixes rather than replacements.

## Discovery and First Run

When the user enables inline suggestions, the next opened tab shows a one-time inline notice:

> AI inline suggestions enabled. Press Tab or RightArrow to accept. Esc to dismiss.

The Settings page labels V1 as **Inline Suggestions (Experimental, StubProvider)** and states that V1 uses canned local suggestions. It differentiates from PSReadLine prediction:

- **PSReadLine prediction**: deterministic history/plugin-based pattern matching inside the shell.
- **Inline suggestions**: a terminal-side preview mechanism intended for future LLM-backed command continuation; V1 validates UX/perf with a stub.

Do not claim parity with any editor product.

## Shell Integration Prerequisite

Inline AI suggestions require OSC 133 prompt marks. Without OSC 133;B prompt-start/edit-line marks, the feature is silent after the one-time prerequisite notice because the controller cannot safely identify the editable command line.

V1 detection and first-run flow:

1. When the user enables `experimental.inlineSuggestions.enabled`, the next prompt starts a 30s detection window.
2. If OSC 133;B is seen within 30s, diagnostics reports shell integration as detected and inline suggestions may trigger normally.
3. Inline Suggestions **REUSES the autofix shell-integration** prerequisite and installer flow; it does not duplicate or fork shell setup. Existing Autofix Settings text confirms enabling Autofix installs OSC 133 shell integration into PowerShell profiles (`src/cascadia/TerminalSettingsEditor/Resources/en-US/Resources.resw:748-750`).
4. If Autofix is already enabled and shell integration is installed, Inline Suggestions just works.
5. If Autofix is not enabled and no OSC 133;B is detected within 30s, show a one-time inline notice in the terminal:

   > Inline Suggestions requires shell integration. Enable Autofix in Settings → AI Agents to install it (one-time setup).

6. Reuse the existing autofix shell-integration check infrastructure and script path handling from `src/cascadia/TerminalApp/AppActionHandlers.cpp:1819-1887`, which embeds the PowerShell OSC 133 shell-integration script. The repo root `shell-integration 1.ps1` shows the same prompt wrapper emitting OSC 133;A, OSC 9;9, OSC 133;B, and OSC 133;D.

Shell support matrix for V1: PowerShell only. Diagnostics reports `Shell integration: not detected` for any shell where OSC 133;B is not observed.

## Visual Spec

```
PS C:\repo> git st█atus --porcelain
            └─┬─┘└─┬─────────────┘
              │    │
              │    └── renderer-native preview text
              └────── what user actually typed
```

- Ghost text begins at the cursor and represents only the suffix.
- It is rendered by the terminal renderer preview path, so it uses the same font, layout, DPI, shaders, and clipping behavior as grid text.
- No box, border, shadow, or separate chrome.
- Single line in V1. Suggestions in V1 are ASCII-only because StubProvider's canned data is ASCII; Unicode is not suppressed by architecture. Invalid content is hidden and diagnostics records `suppressed: <reason>`.

## Interaction Model

### Trigger

1. User types at a prompt.
2. After 300ms of inactivity, if no suppression rule applies, request a `StubProvider` suffix.
3. While fetching, show no spinner and never block typing.
4. V2 may use adaptive debounce: 300ms after a pause longer than 1s and immediate on whitespace token boundary. V1 is fixed 300ms with a config knob.

### While Showing

| User action | Result |
|-------------|--------|
| **Tab** | If no user keybinding and no visible or recently closed shell completion menu, accept. If shell completion is visible or within the 100ms close-transition safety window, send Tab to shell and dismiss AI. |
| **RightArrow at end-of-line** | Accept. |
| **Esc** | Dismiss and suppress for the current line until Enter or Ctrl+C. |
| **Ctrl+Esc** | Pause inline suggestions for this TermControl session until restart. |
| **Command Palette: `togglePauseInlineSuggestions`** | Toggle session pause. |
| **Enter** | Send only typed text; discard ghost text. |
| **Typed character matching first ghost char** | Send the character and shrink the ghost text by one char. |
| **Typed character that diverges, Backspace, arrows, click, focus lost** | Dismiss and restart normal debounce as appropriate. |
| **Ctrl+C** | Pass to PTY, clear suggestion, and clear line-local suppression. |

## Accept-Key Policy

Priority order:

1. **User-defined keybindings** always win.
2. **Tab** while `Showing` accepts only when the shell has no immediate visible tab-completion candidate/menu and no menu was visible within the last 100ms.
3. **Tab** while shell completion is visible or recently closed goes to the shell and AI dismisses.
4. **RightArrow** at end-of-line while `Showing` accepts.
5. Otherwise, default terminal behavior.

Settings use `acceptKeys` plural with default `["tab", "rightArrow"]`.

## PSReadLine Policy

V1 default is `psReadLinePolicy: "aiWinsVisually"`.

When AI preview is showing, `PreviewText` pads to end-of-line and visually covers PSReadLine prediction cells. When AI dismisses, PSReadLine prediction naturally reappears. This avoids fragile detection by color and makes the visual ownership rule explicit.

Optional policies:
- `suppressIfShellPredicting`: hide AI if shell prediction is detected by a reliable signal.
- `aiAlwaysSuppresses`: keep AI hidden whenever the controller believes the shell owns prediction UI.

## Edge Cases and Suppression

| Scenario | Behavior |
|----------|----------|
| Cursor not at end of edit line | Do not show. |
| Inside unmatched single/double quote | Do not show. |
| After `-m`, `--message`, `-c`, or `--body` followed by space + opening quote | Do not show. |
| Right after `|`, `&&`, `||`, or `;` | Do not show until at least 2 chars typed after operator. |
| During bracketed paste or within 500ms after paste | Do not show. |
| Selection active | Do not show. |
| Alt screen buffer | Do not show. |
| No OSC 133;B prompt mark | Do not show. |
| Command running after input-start mark and before a newer prompt mark | Do not show. |
| Suggestion contains newline | Do not show. Unicode code points are allowed; V1 StubProvider suggestions are ASCII-only because the canned provider data is ASCII. |
| Cursor too close to viewport right edge | Do not show. |
| Provider result is not append-only | Reject and do not show. |
| Empty line | Do not show. |

## Defaults

| Setting | Default V1 | Rationale |
|---------|------------|-----------|
| `enabled` | **false** | Experimental and opt-in. |
| `debounceMs` | 300 | Stable fixed V1 delay. |
| `maxLength` | 200 | Keeps suggestions small and viewport-friendly. |
| `acceptKeys` | `["tab", "rightArrow"]` | Tab is familiar; RightArrow avoids completion conflict. |
| `psReadLinePolicy` | `aiWinsVisually` | Clear visual ownership while AI is showing. |
| `provider` | `stub` | V1 only. Real providers are V2/post-V1. |

## Diagnostics

The diagnostics area shows:
- Provider name + status (`StubProvider active` / `unavailable`).
- Shell integration status: `detected` when OSC 133;B has been observed, otherwise `not detected`.
- Last 3 outcomes: `triggered`, `shown`, `stale`, `suppressed: <reason>`, `error`.
- Last suggestion latency.
- **Test inline suggestion** button.

## Telemetry → Privacy

### Provider data flow

V1 ships only local in-proc `StubProvider`. Raw edit text is not sent to another process, service, or model. Real providers are V2 and require privacy review before receiving edit text, cwd, history, or shell context.

### Telemetry

Telemetry captures counts and shapes only:
- Lengths, not text.
- Latencies.
- Accept/dismiss/suppression reasons.
- State transitions.
- Hashed shell name and hashed cwd bucket.

Never capture raw edit text or suggestion text.

## Failure Modes & User-Facing Messaging

| Mode | What user sees |
|------|----------------|
| StubProvider disabled/unavailable | Nothing; diagnostics shows unavailable. |
| StubProvider returns invalid suffix | Nothing; diagnostics records `suppressed: appendOnlyMismatch` or error. |
| Setting enabled but no prompt mark after 30s | One-time inline shell-integration prerequisite notice; after that, no suggestions and diagnostics shows `Shell integration: not detected`. |
| Suppression rule applies (alt buffer, no prompt mark, command running, etc.) | Nothing inline; diagnostics records `suppressed: <reason>`. |
| Future provider transport error | Nothing inline; diagnostics records error. |

No error overlays. Inline suggestions are hidden, not silently disabled: the controller remains alive, suppression records diagnostics, and the feature resumes automatically when the blocking condition clears.

## Accessibility

- AI ghost text is not announced while showing.
- On accept, announce via live region: `Accepted suggestion: <text>`.
- Keyboard-only users have full parity: Tab, RightArrow, Esc, Ctrl+Esc, command-palette pause.
- High-contrast themes rely on the Phase 0.5 `PreviewText` display-attributes extension so InlineSuggestion can request dim/secondary color instead of only inheriting ActionPreview italic styling; verify contrast in Phase 5 accessibility tests.
